# Architecture

## 目标架构

DataPilot 采用前后端分离架构：

- `apps/web`：React Web 应用，负责工作台、SQL 编辑器、连接管理、AI 交互、结果展示和 Admin Console，并作为桌面端共享 UI 源码。
- `apps/api`：Go API 服务，负责数据库连接、SQL 执行、安全审查、AI Provider 调用和审计。
- `apps/desktop`：Wails 桌面端，负责提供丝滑的本地桌面体验，并复用 React 界面。
- `apps/mobile`：后续 Flutter 移动端，当前只作为技术路线预留，不进入 MVP 实现。
- `plugins`：后续新增，用于承载数据库适配器。
- `docs`：产品、架构、贡献和安全文档。

桌面端优先，Web 作为同源部署形态同步跟进。移动端后置，技术栈明确为 Flutter，等桌面端和 Web 核心体验稳定后再启动。

## 后端分层规划

后续建议按以下边界演进：

- `connection`：数据源配置、凭证、连接测试、连接池。
- `database`：数据库元数据、对象浏览、表数据读写。
- `query`：SQL 执行、分页、多结果集、执行历史。
- `safety`：SQL 解析、风险等级、只读策略、二次确认。
- `ai`：AI Provider、上下文构造、提示词模板、调用审计。
- `admin`：管理控制台 API，聚合用户、角色、策略、审计和系统配置。
- `audit`：登录、连接、SQL 执行、导出、AI 请求和危险操作审计。
- `rbac`：用户、角色、权限、数据源授权和操作范围控制。
- `policy`：SQL 风险策略、只读策略、脱敏策略和环境策略。
- `workspace`：团队空间、保存查询、收藏、仪表盘。
- `plugin`：数据库插件注册、能力声明、驱动加载。

API 服务第一阶段优先使用 Go 标准库 `net/http`。如果路由、中间件和鉴权复杂度上来，再评估是否引入 `chi`，暂不引入 `gin` 或大型框架。

## 存储规划

桌面端使用本地 SQLite 保存非敏感元数据：

- 数据源配置。
- 查询历史和保存查询。
- 工作区 Tab、布局和编辑器草稿。
- 用户偏好。
- 本机审计缓存。
- AI 会话索引和摘要。

Web / Server 模式第一阶段也可以使用 SQLite，保持轻量部署；企业部署阶段需要支持 PostgreSQL 作为元数据库。

敏感信息不进入普通 SQLite 表，必须走密钥管理。

## 密钥和凭证管理

桌面端优先使用系统安全存储：

- macOS Keychain。
- Windows Credential Manager。
- Linux Secret Service / libsecret。

Web / Server 模式使用服务端加密存储，主密钥通过环境变量或外部 secret 注入。数据库密码、SSH 私钥、AI API Key、代理认证信息都必须按敏感信息处理。

如果安全存储不可用，必须显式提示用户并使用本地加密文件兜底，不能静默写明文。

## 桌面端规划

桌面端是优先交付形态，从第一天纳入架构，原因是数据库客户端对本地体验要求高：

- 快捷键、窗口、多标签、拖拽和本地文件操作要顺滑。
- 本地连接数据库、SSH 隧道和驱动管理更自然。
- 可以离线使用本地模型或本地配置。
- Go + Wails 的体积和内存占用明显低于 Electron。

桌面端优先复用 `apps/web` 的 React 组件。只有本地能力通过 Wails 的 Go binding 暴露，不在前端硬编码平台差异。

桌面优先阶段必须先跑通：

- Wails dev/build 链路。
- 共享 React UI 到桌面窗口。
- 深色模式和基础窗口体验。
- Command Palette 和全局快捷键。
- 多标签工作区。
- 本地偏好设置。
- 工作区恢复。

## Admin Console 规划

Admin Console 是企业审计和治理入口，优先作为 `apps/web` 的 `/admin` 路由实现，不单独创建 `apps/admin`。后续只有当管理端复杂度明显上升时，才考虑拆分独立应用。

第一阶段重点不是做传统后台，而是把审计和策略能力从底层设计好：

- 用户、角色、权限和数据源授权。
- 数据源治理、密钥配置、只读策略和脱敏策略。
- SQL 执行审计、导出审计、连接审计和危险操作确认记录。
- AI 请求审计，包括模型、上下文摘要、生成 SQL、用户确认和执行结果关联。
- SQL 风险策略，例如禁止 `DROP`、限制无 `WHERE` 的 `UPDATE`/`DELETE`、生产环境默认只读。
- 系统配置，包括 AI Provider、代理、日志保留周期和审计导出。

桌面端可以展示个人审计和本机配置，但企业级治理入口默认放在 Web Admin Console。

## 移动端规划

移动端使用 Flutter，不复用 React 代码。原因是移动端需要更自然的手势、离线体验、系统能力和原生性能。

移动端不进入第一阶段 MVP，后续优先做：

- 连接列表和只读查询。
- 查询历史、收藏和结果查看。
- AI SQL 助手。
- 移动端安全限制，默认不执行高风险 DDL/DML。

## 插件化方向

每个数据库插件应该声明：

- 数据库类型和版本范围。
- 连接参数模型。
- 驱动依赖和版本。
- 元数据读取能力。
- SQL 方言能力。
- DDL 生成能力。
- 查询计划能力。
- 数据导入导出能力。
- 安全能力限制。

核心服务只依赖统一接口，不依赖具体数据库实现。

第一阶段内置 MySQL、PostgreSQL、SQLite。插件市场、远程驱动下载和第三方插件签名后置。

## 备份和迁移

桌面端必须支持配置资产迁移：

- 导出连接配置，不包含密钥明文。
- 导出查询历史和保存查询。
- 导出用户偏好。
- 导入配置包。

本地 SQLite schema、配置文件和插件能力声明都必须带版本并支持迁移。重大升级前需要备份本地元数据。

## AI 安全原则

- AI 只能生成建议，不能绕过权限直接执行危险 SQL。
- 所有 AI 生成 SQL 必须展示给用户确认。
- 默认提供只读连接模式。
- DDL、DELETE、UPDATE、TRUNCATE、DROP 等高风险操作必须标记。
- AI 上下文默认最小化，只发送必要的 schema 信息。
- 后续支持脱敏和本地模型模式。

## 与 Chat2DB 的关系

`Chat2DB` 是参考对象，主要参考其产品范围、前后端形态和数据库插件思路。本项目从空目录开始实现，不直接复制其代码、资源或许可证文本。
